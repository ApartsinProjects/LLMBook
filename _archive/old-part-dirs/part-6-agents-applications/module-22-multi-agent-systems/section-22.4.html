Production Multi-Agent Systems

A demo multi-agent system costs ten dollars per run. A production multi-agent system must cost ten cents, or it will never ship.

Census AI Agent

Prerequisites

This section builds on multi-agent architecture patterns from Section 22.2: Multi-Agent Architecture Patterns, agentic workflow engineering from Section 22.3: Agentic Workflows & Pipelines, and the production API cost controls covered in Section 9.3.

Key Insight: The Restaurant Kitchen

A production multi-agent system is like a commercial restaurant kitchen, not a home cooking experiment. In your home kitchen (prototype), you can taste as you go, redo dishes, and spend whatever you want on ingredients. In a commercial kitchen (production), every ingredient has a cost, every station has a time budget, the head chef (orchestrator) tracks every plate, health inspectors (observability) monitor everything, and when someone drops a dish (failure), the kitchen has a recovery protocol that does not require restarting the entire service. The gap between a working demo and a production system is the same gap between cooking dinner for friends and running a restaurant that serves 500 covers per night.

Big Picture

Most multi-agent systems never make it past the demo stage because they are too expensive, too fragile, and too opaque for production. A research pipeline that spawns five agents per query, each making multiple LLM calls, can easily cost $1-5 per request. At 10,000 requests per day, that is $10,000-50,000 per month in LLM costs alone. This section covers the engineering practices that make multi-agent systems viable in production: cost budgets that prevent runaway spending, the A2A (Agent-to-Agent) protocol for standardized inter-agent communication, testing strategies that catch failures before users do, scaling patterns that handle load spikes, and observability that makes the system debuggable. The error handling and retry patterns from Section 9.2 form the foundation for multi-agent reliability.

1. Cost Management for Multi-Agent Systems

Multi-agent systems multiply costs in ways that single-agent systems do not. Each agent in a pipeline makes its own LLM calls, and those calls compound. A supervisor agent calls a planner, the planner delegates to three workers, each worker makes two tool calls that require LLM interpretation, and the supervisor reviews the combined output. That is 10+ LLM calls for a single user query. Without explicit cost controls, these systems become prohibitively expensive at scale.

Budget Enforcement Architecture

The most effective cost control pattern is a hierarchical budget system where the orchestrator allocates a token budget to each sub-agent, and each sub-agent enforces its own limit. If any agent exhausts its budget, it must return a partial result rather than making additional calls. This prevents runaway cost scenarios where a single confused agent makes dozens of LLM calls trying to recover from an error. Code Fragment 22.4.1 below puts this into practice.


# Define AgentBudget, SystemBudget; implement check_and_record, allocate
# Key operations: agent orchestration
from dataclasses import dataclass, field
from typing import Optional
import threading

@dataclass
class AgentBudget:
    """Enforces cost limits for a single agent within a multi-agent system."""
    max_input_tokens: int = 50_000
    max_output_tokens: int = 10_000
    max_llm_calls: int = 5
    max_cost_usd: float = 0.50

    # Running totals
    _input_tokens: int = field(default=0, init=False)
    _output_tokens: int = field(default=0, init=False)
    _llm_calls: int = field(default=0, init=False)
    _cost_usd: float = field(default=0.0, init=False)
    _lock: threading.Lock = field(default_factory=threading.Lock, init=False)

    def check_and_record(self, input_tokens: int, output_tokens: int,
                          cost_usd: float) -> bool:
        """Record usage and return False if budget would be exceeded."""
        with self._lock:
            new_input = self._input_tokens + input_tokens
            new_output = self._output_tokens + output_tokens
            new_cost = self._cost_usd + cost_usd
            new_calls = self._llm_calls + 1

            if (new_input > self.max_input_tokens or
                new_output > self.max_output_tokens or
                new_cost > self.max_cost_usd or
                new_calls > self.max_llm_calls):
                return False

            self._input_tokens = new_input
            self._output_tokens = new_output
            self._cost_usd = new_cost
            self._llm_calls = new_calls
            return True

@dataclass
class SystemBudget:
    """Top-level budget that distributes across agents."""
    total_budget_usd: float = 2.00

    def allocate(self, agent_name: str, fraction: float) -> AgentBudget:
        """Give an agent a fraction of the total budget."""
        return AgentBudget(max_cost_usd=self.total_budget_usd * fraction)

# Example: allocate budget across a 3-agent pipeline
system = SystemBudget(total_budget_usd=1.00)
researcher_budget = system.allocate("researcher", fraction=0.40)
writer_budget = system.allocate("writer", fraction=0.35)
reviewer_budget = system.allocate("reviewer", fraction=0.25)
Code Fragment 22.4.1: A hierarchical budget system that tracks token usage per agent and enforces limits. When a sub-agent exhausts its allocation, it returns a partial result rather than making additional calls, preventing runaway costs.

2. The A2A (Agent-to-Agent) Protocol

Google's Agent-to-Agent (A2A) protocol, introduced in April 2025, provides a standardized way for agents built by different teams (or even different organizations) to discover each other's capabilities, negotiate communication formats, and exchange messages. Before A2A, every multi-agent system invented its own inter-agent communication format, making it impossible to compose agents from different frameworks.

Core Concepts

Before starting this section, you should be comfortable with Agent Card: A JSON document (hosted at /.well-known/agent.json) that describes an agent's capabilities, accepted input formats, authentication requirements, and endpoint URL. Think of it as an API specification for agents., Task: The fundamental unit of work in A2A. A client agent creates a task, sends it to a remote agent, and receives status updates and results. Tasks have lifecycle states: submitted, working, completed, failed, or canceled., Message and Part: Communication happens through messages containing typed parts (text, file, structured data). This allows agents to exchange rich content including images, documents, and JSON payloads., and Streaming: For long-running tasks, A2A supports Server-Sent Events (SSE) so the client agent can receive incremental progress updates rather than waiting for the full result.. Code Fragment 22.4.2 below puts this into practice.


# Define A2AClient; implement __init__
# Key operations: results display, agent orchestration, API interaction
import httpx
import json

class A2AClient:
    """Minimal A2A protocol client for agent-to-agent communication."""

    def __init__(self, agent_url: str):
        self.agent_url = agent_url.rstrip("/")
        self.client = httpx.AsyncClient(timeout=120)

    async def discover(self) -> dict:
        """Fetch the remote agent's Agent Card."""
        resp = await self.client.get(
            f"{self.agent_url}/.well-known/agent.json"
        )
        return resp.json()

    async def send_task(self, message: str, task_id: str = None) -> dict:
        """Send a task to the remote agent following A2A spec."""
        payload = {
            "jsonrpc": "2.0",
            "method": "tasks/send",
            "params": {
                "id": task_id or generate_task_id(),
                "message": {
                    "role": "user",
                    "parts": [{"type": "text", "text": message}]
                }
            }
        }
        resp = await self.client.post(self.agent_url, json=payload)
        return resp.json()

# Usage: discover a remote research agent, then send it a task
async def delegate_research(topic: str):
    research_agent = A2AClient("https://research-agent.internal/a2a")

    # Step 1: Discover capabilities
    card = await research_agent.discover()
    print(f"Agent: {card['name']}, Skills: {card['skills']}")

    # Step 2: Send task
    result = await research_agent.send_task(
        f"Research the following topic and return a summary: {topic}"
    )
    return result["result"]["status"]
Code Fragment 22.4.2: A minimal A2A protocol client that discovers remote agents via their agent cards, sends tasks using JSON-RPC, and streams progress updates through Server-Sent Events.
Note

A2A and MCP (Model Context Protocol) solve different problems. MCP standardizes how agents connect to tools and data sources (databases, APIs, file systems). A2A standardizes how agents communicate with other agents. In a production system, an individual agent uses MCP to access its tools, and A2A to delegate subtasks to other agents. The two protocols are complementary, not competing. See Section 21.2 for MCP details.

3. Testing Multi-Agent Systems

Testing multi-agent systems is fundamentally harder than testing single-agent systems because the interactions between agents create emergent behaviors that cannot be predicted from testing each agent in isolation. A researcher agent and a writer agent might each work perfectly alone, but when composed, the researcher might produce output in a format the writer cannot parse. The solution is a layered testing strategy. Code Fragment 22.4.3 below puts this into practice.

The Testing Pyramid for Multi-Agent Systems

  • Unit tests (per agent): Test each agent in isolation with mocked tool responses and fixed LLM outputs. Verify that the agent correctly parses inputs, calls the right tools, and formats outputs according to its contract.
  • Contract tests (agent boundaries): Verify that the output schema of one agent matches the input schema expected by the next agent in the pipeline. These tests catch the most common integration failures.
  • Integration tests (agent pairs): Run two adjacent agents together with real LLM calls but controlled inputs. Verify end-to-end data flow and error propagation between agents.
  • System tests (full pipeline): Run the complete multi-agent system against a suite of representative tasks. Measure accuracy, cost, latency, and failure rates across the full pipeline.

# implement test_researcher_output_matches_writer_input_contract
# Key operations: RAG pipeline, agent orchestration
import pytest
from unittest.mock import AsyncMock, patch

# Contract test: verify researcher output matches writer input schema
def test_researcher_output_matches_writer_input_contract():
    """The researcher must produce output the writer can consume."""
    researcher_output = {
        "topic": "quantum computing advances",
        "sources": [
            {"title": "Paper A", "summary": "...", "url": "https://..."},
        ],
        "key_findings": ["Finding 1", "Finding 2"],
    }

    # Writer agent expects these fields
    required_fields = {"topic", "sources", "key_findings"}
    assert required_fields.issubset(researcher_output.keys())

    # Each source must have title and summary
    for source in researcher_output["sources"]:
        assert "title" in source
        assert "summary" in source

# Integration test: researcher + writer with real LLM calls
@pytest.mark.asyncio
async def test_researcher_writer_integration():
    """Two agents work together on a controlled input."""
    researcher = ResearchAgent(budget=AgentBudget(max_cost_usd=0.20))
    writer = WriterAgent(budget=AgentBudget(max_cost_usd=0.15))

    # Run researcher on known topic
    research = await researcher.run("Summarize recent CRISPR advances")
    assert research.status == "completed"
    assert len(research.sources) >= 1

    # Feed researcher output to writer
    article = await writer.run(research.to_dict())
    assert article.status == "completed"
    assert len(article.text) > 500  # Reasonable article length

4. Scaling Patterns

Multi-agent systems face unique scaling challenges. Unlike a single API endpoint where you can simply add more replicas behind a load balancer, multi-agent workflows involve stateful conversations, inter-agent dependencies, and variable execution times. Three patterns address these challenges.

Pattern 1: Queue-Based Decoupling

Place a message queue (Redis Streams, RabbitMQ, or Amazon SQS) between agents so that each agent processes tasks independently at its own rate. The researcher agent publishes completed research to a queue; the writer agent consumes from that queue whenever it is ready. This decouples agents that operate at different speeds and provides natural backpressure when downstream agents cannot keep up.

Pattern 2: Worker Pools with Concurrency Limits

Run multiple instances of each agent type as a worker pool, with a concurrency limiter that prevents any single agent type from consuming too many LLM API tokens simultaneously. This prevents a burst of researcher tasks from starving writer agents of API capacity.

Pattern 3: Circuit Breakers per Agent

Wrap each agent in a circuit breaker (as described in Section 9.3) that opens when the agent's error rate exceeds a threshold. When the circuit opens, the orchestrator can route tasks to a fallback agent or return a degraded response rather than cascading failures through the entire pipeline.

5. Observability for Multi-Agent Workflows

When a multi-agent system produces a bad result, the first question is always "which agent caused the problem?" Without proper observability, answering this question requires manually reading through logs from every agent in the pipeline. With proper observability, you can trace a single request through every agent, see exactly what each agent received, what it decided, and what it produced.

Distributed Tracing with OpenTelemetry

The standard approach is to propagate a trace ID through the entire multi-agent pipeline. Each agent creates a span within that trace, recording its inputs, outputs, LLM calls, tool invocations, and timing. Tools like Langfuse, LangSmith, and Arize Phoenix provide specialized UIs for visualizing multi-agent traces that show the complete agent interaction tree.

6. Failure Modes and Recovery

Multi-agent systems have failure modes that do not exist in single-agent systems. Understanding these modes and building recovery mechanisms for each is what separates production systems from demos.

6. Failure Modes and Recovery Comparison
Failure Mode Description Recovery Strategy
Cascading hallucination Agent A hallucinates facts; Agent B builds on them; Agent C cites them as verified Cross-verification gates between agents; require source citations
Budget exhaustion An early agent consumes the entire budget, leaving nothing for later stages Per-agent budget caps with graceful degradation
Infinite delegation loop Agent A delegates to Agent B, which delegates back to Agent A Maximum delegation depth; cycle detection in the orchestrator
Schema mismatch One agent's output format changes subtly, breaking the downstream agent Schema validation at agent boundaries; contract tests
Partial failure One of three parallel workers fails; the others succeed Collect available results; retry failed worker or proceed with partial data
Timeout cascade A slow agent causes the orchestrator to timeout, which causes the client to timeout Per-agent timeouts shorter than the orchestrator timeout; early termination
Warning

The most dangerous failure mode in multi-agent systems is cascading hallucination, where one agent fabricates information and downstream agents treat it as fact. Unlike a single-agent system where a human can spot the hallucination in the output, multi-agent pipelines can launder hallucinations through multiple processing stages until they appear well-sourced and credible. Always include a verification step that checks critical facts against ground-truth sources, and never let an agent cite another agent as a primary source.

Fun Fact

A team at a major tech company built a multi-agent content pipeline where a researcher found articles, a writer summarized them, and a fact-checker verified the summaries. During testing, they discovered the fact-checker had a subtle bug: it was verifying summaries against the researcher's notes rather than the original sources. Since the researcher had already hallucinated several facts, the fact-checker happily confirmed them all as accurate. The team renamed the pipeline "the echo chamber" and added ground-truth verification before shipping.

Multi-Agent Compliance Review Pipeline at a Financial Institution

Who: Platform engineering team at a mid-size investment firm

Situation: Regulatory filings required cross-checking data from multiple internal systems, verifying calculations, and producing audit-ready reports, all within tight quarterly deadlines.

Problem: A single-agent approach hallucinated regulatory references and mixed up calculation methodologies. Manual review caught the errors, but the review cycle consumed most of the time savings.

Decision: The team deployed a four-agent pipeline: a data extraction agent (with per-agent budget caps), a calculation verification agent, a compliance cross-referencing agent, and a report drafting agent. Each agent had strict schema contracts validated by automated contract tests (see Section 3 above), and the pipeline used queue-based decoupling with circuit breakers per agent (see Section 9.3).

Result: Cascading hallucinations dropped to near zero because each agent verified facts against original source databases rather than prior agent outputs. Per-request cost fell from $3.20 to $0.45 through hierarchical budget allocation and model tiering (a smaller model for extraction, a larger model for compliance reasoning). Quarterly filing time decreased from 12 days to 3 days.

Lesson: Production multi-agent reliability depends on combining budget caps, contract testing, ground-truth verification, and agent-level observability; no single technique is sufficient on its own.

📝 Knowledge Check

1. Why is hierarchical budget allocation more effective than a single global budget for multi-agent systems?
Show Answer
A single global budget allows a misbehaving agent to consume the entire budget before downstream agents get to run. Hierarchical allocation gives each agent its own cap, ensuring that one agent's overspending cannot starve the rest of the pipeline. It also makes it easy to identify which agent is the most expensive and optimize accordingly.
2. What is the difference between A2A and MCP?
Show Answer
MCP (Model Context Protocol) standardizes how agents connect to tools and data sources (databases, APIs, file systems). A2A (Agent-to-Agent) standardizes how agents communicate with other agents. MCP is for agent-to-tool connections; A2A is for agent-to-agent delegation. They are complementary protocols used together in production multi-agent systems.
3. What is cascading hallucination, and how do you prevent it?
Show Answer
Cascading hallucination occurs when Agent A fabricates information, Agent B processes and builds on that fabrication, and Agent C cites the processed version as verified truth. Each stage launders the hallucination, making it harder to detect. Prevention requires cross-verification gates that check facts against original sources (not other agents' outputs), mandatory source citations that trace back to ground-truth data, and contract tests that validate the factual accuracy of inter-agent messages.
Research Frontier

Open Questions:

  • How should we price multi-agent systems for end users? Per-request pricing is simple but unpredictable when different requests trigger vastly different agent configurations. Value-based pricing is more aligned but harder to implement.
  • Can multi-agent systems self-optimize their topology? Research on "AutoAgents" suggests that agents can discover the optimal pipeline configuration for a given task class, but this remains early-stage.

Recent Developments (2024-2025):

  • Google's A2A protocol (April 2025) established the first industry standard for agent-to-agent communication, with support from over 50 technology partners at launch.
  • OpenAI's Agent SDK and Anthropic's Agent SDK (2025) both include built-in support for multi-agent orchestration, cost tracking, and distributed tracing, significantly lowering the barrier to production multi-agent systems.

Explore Further: Instrument a multi-agent pipeline with OpenTelemetry traces. Analyze the trace data to identify which agent is the cost and latency bottleneck, then experiment with replacing that agent with a cheaper model or caching its outputs.

Key Takeaways

  • Cost is the primary barrier to production multi-agent systems. Implement hierarchical budget allocation with per-agent caps and total-system limits. Monitor cost per request continuously.
  • The A2A protocol standardizes agent-to-agent communication. Use Agent Cards for capability discovery, JSON-RPC for task dispatch, and SSE for streaming progress updates. A2A complements MCP (which connects agents to tools).
  • Test multi-agent systems in layers. Unit tests per agent, contract tests at boundaries, integration tests for agent pairs, and system tests for the full pipeline. Contract tests catch the most common production failures.
  • Scale with queue-based decoupling and circuit breakers. Place message queues between agents for backpressure, use worker pools with concurrency limits, and wrap each agent in a circuit breaker to prevent cascading failures.
  • Observability requires distributed tracing. Propagate trace IDs through the pipeline so you can identify which agent caused a bad output. Use specialized tools (Langfuse, LangSmith) for multi-agent trace visualization.
  • Guard against cascading hallucination. Verify facts against ground-truth sources, not other agents' outputs. Include explicit verification gates between pipeline stages.
