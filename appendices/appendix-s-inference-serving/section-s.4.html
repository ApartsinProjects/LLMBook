Quantization for Serving: GPTQ, AWQ, and GGUF

Big Picture

Quantization reduces model weights from 16-bit or 32-bit floating point to lower precision formats (8-bit, 4-bit, or even 2-bit integers), dramatically shrinking memory requirements and often increasing inference speed. This section compares the four most widely used quantization methods for LLM serving: GPTQ, AWQ, GGUF, and bitsandbytes. Each method makes different tradeoffs between quality, speed, and ease of use, and each integrates differently with the serving frameworks covered in previous sections.

1. Why Quantize for Serving?

A 70-billion parameter model stored in FP16 requires approximately 140 GB of GPU memory just for the weights. That exceeds the capacity of any single consumer GPU and requires at least two A100-80GB cards. Quantization to 4-bit precision reduces this to roughly 35 GB, fitting on a single GPU. Beyond memory savings, quantized models benefit from higher memory bandwidth utilization: since each token generation step is memory-bound (not compute-bound), moving fewer bytes per parameter directly translates to more tokens per second.

The key question is how much quality is lost. Modern quantization methods are remarkably effective at preserving model quality. A well-quantized 4-bit model typically scores within 1 to 3 percentage points of the original FP16 model on standard benchmarks, while using 4x less memory and often running 2x faster.

Key Insight

LLM inference is memory-bandwidth bound, not compute-bound. During token generation, each parameter is read once per token but participates in only a single multiply-accumulate operation. The GPU's computational throughput is vastly underutilized while it waits for weights to arrive from memory. Quantization helps because it reduces the number of bytes transferred per parameter, directly increasing the effective memory bandwidth.

2. GPTQ: Post-Training Quantization with Calibration

GPTQ (Generative Pre-Trained Transformer Quantization) is a one-shot, weight-only quantization method that uses a small calibration dataset to minimize the quantization error layer by layer. It works by solving an optimization problem: for each layer, find the quantized weights that minimize the difference between the original and quantized layer outputs on the calibration data.

GPTQ quantizes weights to 4-bit or 8-bit integers while keeping activations in FP16. The calibration process typically requires 128 to 256 samples and takes 1 to 4 hours depending on model size. Once quantized, the model can be loaded instantly without further calibration.

The following example demonstrates quantizing a model using the auto-gptq library. 

from transformers import AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

model_id = "meta-llama/Llama-3.1-8B-Instruct"
quant_output = "./llama-3.1-8b-gptq-4bit"

# Define quantization configuration
quantize_config = BaseQuantizeConfig(
    bits=4,              # 4-bit quantization
    group_size=128,      # Quantize in groups of 128 weights
    desc_act=True,       # Use activation ordering for better quality
    sym=False,           # Asymmetric quantization
)

# Load model and tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoGPTQForCausalLM.from_pretrained(
    model_id,
    quantize_config=quantize_config,
)

# Prepare calibration dataset (128 samples from wikitext)
from datasets import load_dataset
calib_data = load_dataset("wikitext", "wikitext-2-raw-v1", split="train")
calib_texts = [tokenizer(t["text"], return_tensors="pt", max_length=512, truncation=True)
               for t in calib_data.select(range(128)) if len(t["text"]) > 50]

# Quantize (takes 1-4 hours for 70B models)
model.quantize(calib_texts)

# Save the quantized model
model.save_quantized(quant_output)
tokenizer.save_pretrained(quant_output)

2.1 Serving GPTQ Models

Both vLLM and TGI natively support GPTQ models. Simply point the server at a GPTQ-quantized model and specify the quantization format. 

# Serve a GPTQ model with vLLM
vllm serve TheBloke/Llama-2-70B-Chat-GPTQ \
    --quantization gptq \
    --tensor-parallel-size 2 \
    --max-model-len 4096

# Serve a GPTQ model with TGI
docker run --gpus all --shm-size 1g -p 8080:80 \
    -v /data:/data \
    ghcr.io/huggingface/text-generation-inference:latest \
    --model-id TheBloke/Llama-2-70B-Chat-GPTQ \
    --quantize gptq

3. AWQ: Activation-Aware Weight Quantization

AWQ (Activation-Aware Weight Quantization) takes a different approach from GPTQ. Instead of minimizing layer-wise reconstruction error, AWQ observes that a small fraction of weights (roughly 1%) are disproportionately important because they correspond to large activation magnitudes. AWQ protects these salient weights by scaling them up before quantization and scaling them back down during inference. This per-channel scaling approach is simpler than GPTQ's optimization and often produces comparable or better quality.

The following example shows how to quantize a model using the autoawq library. 

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_id = "meta-llama/Llama-3.1-8B-Instruct"
quant_output = "./llama-3.1-8b-awq-4bit"

# Load model
model = AutoAWQForCausalLM.from_pretrained(model_id)
tokenizer = AutoTokenizer.from_pretrained(model_id)

# Quantize with AWQ (faster than GPTQ, typically 30-60 minutes for 70B)
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM",  # Use GEMM kernel for serving (vs. GEMV for batch-1)
}
model.quantize(tokenizer, quant_config=quant_config)

# Save
model.save_quantized(quant_output)
tokenizer.save_pretrained(quant_output)
Tip

AWQ with the GEMM kernel version is generally recommended for serving scenarios where batch sizes exceed 1. The GEMV version is optimized for single-request (batch size 1) latency. If your workload mixes both, GEMM is the safer default.

4. GGUF and llama.cpp: CPU-Friendly Quantization

GGUF (GPT-Generated Unified Format) is the file format used by llama.cpp, the popular C/C++ inference engine that runs LLMs on CPUs, Apple Silicon, and consumer GPUs. Unlike GPTQ and AWQ, which target NVIDIA GPUs and CUDA kernels, GGUF is designed for portability. It supports a wide range of quantization levels and can split computation between CPU and GPU.

GGUF supports quantization types ranging from Q2_K (roughly 2.5 bits per weight) to Q8_0 (8 bits per weight). The naming convention indicates the bit width and the quantization scheme. The most commonly used levels for serving are Q4_K_M and Q5_K_M, which balance quality and size.

To convert a HuggingFace model to GGUF format, use the conversion script included with llama.cpp. 

# Clone llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp

# Build (with CUDA support for GPU offloading)
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release

# Convert a HuggingFace model to GGUF
python convert_hf_to_gguf.py \
    /path/to/meta-llama/Llama-3.1-8B-Instruct \
    --outfile llama-3.1-8b.gguf \
    --outtype f16

# Quantize to Q4_K_M
./build/bin/llama-quantize llama-3.1-8b.gguf llama-3.1-8b-Q4_K_M.gguf Q4_K_M

4.1 Running a GGUF Server

llama.cpp includes a built-in HTTP server with an OpenAI-compatible API. This makes it a viable serving option for environments without NVIDIA GPUs. 

# Start the llama.cpp server with GPU offloading
./build/bin/llama-server \
    -m llama-3.1-8b-Q4_K_M.gguf \
    --host 0.0.0.0 \
    --port 8080 \
    -ngl 99 \
    -c 4096 \
    --threads 8

The -ngl 99 flag offloads all layers to GPU. On systems without a GPU, omit this flag and llama.cpp will run entirely on CPU, using SIMD optimizations (AVX2, AVX-512, ARM NEON) for acceptable performance on modern processors.

5. bitsandbytes: On-the-Fly Quantization

Unlike the previous methods, which require a separate quantization step, bitsandbytes quantizes model weights during loading. This is the simplest approach: you load any HuggingFace model with a single flag, and it is automatically quantized to 4-bit or 8-bit precision using the NF4 (NormalFloat4) or INT8 data types. 

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

quantization_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",           # NormalFloat4 data type
    bnb_4bit_compute_dtype="float16",     # Compute in FP16
    bnb_4bit_use_double_quant=True,       # Quantize the quantization constants too
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B-Instruct",
    quantization_config=quantization_config,
    device_map="auto",
)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.1-8B-Instruct")

inputs = tokenizer("The capital of France is", return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, max_new_tokens=50)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
Warning

bitsandbytes quantization is convenient for development and prototyping, but it produces lower throughput than GPTQ or AWQ for serving. The lack of optimized CUDA kernels for the NF4 format means that decoding speed is typically 30% to 50% slower than a pre-quantized GPTQ model at the same bit width. For production serving, prefer GPTQ or AWQ.

Practical Example

A common production pattern is to maintain two quantized versions of the same model: an AWQ version for your GPU-based serving cluster (using vLLM or TGI) and a Q4_K_M GGUF version for local developer testing (using llama.cpp or Ollama). The quality difference between these formats at 4-bit is typically less than 1% on standard benchmarks, ensuring consistent behavior between development and production.

Summary

Quantization is an essential technique for making LLM serving practical and affordable. GPTQ and AWQ provide the highest throughput on NVIDIA GPUs through optimized CUDA kernels, making them ideal for production serving with vLLM and TGI. GGUF and llama.cpp open the door to CPU and edge deployment with remarkable efficiency. bitsandbytes offers the lowest barrier to entry for experimentation.
