B.1 Learning Paradigms

Machine learning methods are traditionally grouped by the type of supervision they receive during training. Understanding these categories helps you choose the right approach for a given problem and recognize how LLMs blur the boundaries between them.

Supervised Learning

In supervised learning, the model trains on labeled examples: each input x is paired with a target y. The model learns to map inputs to targets by minimizing a loss function that measures prediction errors.

  • Classification: The target is a category. Examples include sentiment analysis (positive/negative), spam detection, and named entity recognition.
  • Regression: The target is a continuous number. Examples include predicting model latency or estimating document relevance scores.

Fine-tuning an LLM on labeled data (Chapter 13) is supervised learning. The input is a prompt, and the target is the desired completion.

Unsupervised Learning

Unsupervised learning finds structure in data without labels. Clustering (grouping similar items), dimensionality reduction (compressing representations), and density estimation (modeling data distributions) all fall under this umbrella.

LLM pretraining is often described as "self-supervised" learning: the labels come from the data itself. Given a sequence of tokens, the model predicts the next token. No human annotator provides labels; the training signal is extracted directly from the structure of language.

Reinforcement Learning

In reinforcement learning (RL), an agent takes actions in an environment and receives rewards. The goal is to learn a policy that maximizes cumulative reward. RLHF (Reinforcement Learning from Human Feedback), covered in Chapter 16, uses RL to align LLM outputs with human preferences.

Key Insight: LLMs Span All Three Paradigms

A modern LLM's lifecycle touches all three learning paradigms. Pretraining is self-supervised (predict the next token). Supervised fine-tuning (SFT) uses labeled instruction-response pairs. RLHF/DPO applies reinforcement learning with human preference signals. Each stage contributes different capabilities, and understanding the paradigm helps you understand what each stage optimizes for.

B.2 Loss Functions and Optimization

A loss function quantifies how wrong the model's predictions are. Training is the process of minimizing this function. The choice of loss function shapes what the model learns, so choosing wisely is essential.

Common Loss Functions

Common Loss Functions Comparison
Loss Function Formula Use Case
Cross-Entropy $- \Sigma y_i \log(p_i)$ Language modeling, classification
Mean Squared Error $(1/n) \Sigma (y_i - p_i)^2$ Regression, reward modeling
Binary Cross-Entropy $-[y \log(p) + (1-y) \log(1-p)]$ Binary classification, DPO preference pairs
Contrastive Loss Various formulations Embedding training (Chapter 18), CLIP
Hinge Loss $\max(0, 1 - y \cdot p)$ SVMs, ranking tasks

Optimizers

Gradient descent is the foundation, but modern training uses more sophisticated optimizers that adapt learning rates per parameter: Code Fragment B.1 below puts this into practice.

  • SGD (Stochastic Gradient Descent): Updates weights using the gradient from a mini-batch of data. Simple but requires careful learning rate tuning.
  • Adam: Maintains running averages of both the gradient (first moment) and the squared gradient (second moment). This adapts the learning rate for each parameter individually. Adam is the default optimizer for most LLM training.
  • AdamW: A variant of Adam with decoupled weight decay regularization. Used in training BERT, GPT, and most modern LLMs.
  • Adafactor: A memory-efficient alternative to Adam that factorizes the second-moment estimates. Used when training very large models where Adam's memory overhead is prohibitive.

# PyTorch implementation
# Key operations: forward pass computation, loss calculation, training loop
import torch

model = ...  # your neural network

# AdamW with a typical LLM learning rate and weight decay
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=2e-5,            # learning rate
    weight_decay=0.01,   # L2 regularization
    betas=(0.9, 0.999),  # momentum parameters
)

# Training loop skeleton
for batch in dataloader:
    loss = model(batch)       # forward pass
    loss.backward()           # compute gradients
    optimizer.step()          # update weights
    optimizer.zero_grad()     # reset gradients

Learning Rate Scheduling

Most LLM training runs use a warmup phase (learning rate increases linearly from 0 to its peak over the first few thousand steps) followed by a cosine decay (learning rate decreases smoothly back toward 0). This schedule prevents early instability and allows fine-grained convergence later in training.

B.3 Overfitting, Regularization, and Validation

Overfitting occurs when a model memorizes the training data instead of learning general patterns. The model performs well on training examples but poorly on new, unseen data. This is one of the most fundamental challenges in machine learning, and it manifests differently in the LLM world.

Signs of Overfitting

  • Training loss continues to decrease while validation loss increases.
  • The model reproduces training examples verbatim instead of generalizing.
  • Performance on held-out benchmarks plateaus or degrades after a certain point in training.

Regularization Techniques

Regularization Techniques Comparison
Technique How It Works LLM Usage
Dropout Randomly zeroes a fraction of activations during training Used in BERT; less common in modern autoregressive LLMs
Weight Decay Adds a penalty proportional to weight magnitude to the loss Standard in all LLM training (via AdamW)
Early Stopping Stop training when validation performance stops improving Common in fine-tuning; pretraining usually runs to a compute budget
Data Augmentation Create synthetic training examples by transforming existing ones Paraphrasing, back-translation, synthetic data (Chapter 12)

Train / Validation / Test Splits

The standard practice is to split your data into three partitions:

  • Training set (~80%): The model learns from this data.
  • Validation set (~10%): Used to tune hyperparameters and detect overfitting during training.
  • Test set (~10%): Evaluated only once, after all decisions are finalized. This gives an unbiased estimate of real-world performance.

Cross-validation (k-fold) is an alternative when data is scarce: the data is split into k folds, and the model is trained k times, each time using a different fold as validation. This is less common for LLMs (due to the computational cost) but is standard practice for smaller ML models and for evaluating fine-tuned classifiers.

Warning: Data Contamination in LLMs

Because LLMs are pretrained on massive internet corpora, there is a risk that test set examples appeared in the pretraining data. This is called data contamination, and it can artificially inflate benchmark scores. Always check for contamination when evaluating, and prefer held-out or recently created benchmarks that could not have appeared in the training data.

B.4 Evaluation Metrics

Choosing the right metric is as important as choosing the right model. Different tasks require different measurements, and a single number rarely tells the whole story.

Classification Metrics

  • Accuracy: Fraction of correct predictions. Simple but misleading for imbalanced datasets. If 99% of emails are not spam, a model that always predicts "not spam" achieves 99% accuracy while being completely useless.
  • Precision: Of all items the model labeled positive, what fraction are truly positive? High precision means few false positives.
  • Recall: Of all truly positive items, what fraction did the model find? High recall means few false negatives.
  • F1 Score: The harmonic mean of precision and recall: $F1 = 2 \cdot (P \cdot R) / (P + R)$. Useful when you want a single number that balances both concerns.

Language Generation Metrics

Language Generation Metrics Comparison
Metric Measures Range Used For
Perplexity How surprised the model is by the data 1 to ∞ (lower is better) Language model evaluation
BLEU N-gram overlap with reference text 0 to 100 (higher is better) Machine translation
ROUGE Recall-oriented n-gram overlap 0 to 1 (higher is better) Summarization
BERTScore Semantic similarity via embeddings 0 to 1 (higher is better) General text generation
METEOR Alignment with synonyms and stemming 0 to 1 (higher is better) Machine translation, captioning
Key Insight: Automated Metrics Have Limits

BLEU and ROUGE measure surface-level text overlap, not meaning. A paraphrase that perfectly captures the intended meaning but uses different words will score poorly. This is why the field is increasingly moving toward LLM-as-judge evaluation (Chapter 25), where a powerful language model rates the quality of generated text. Human evaluation remains the gold standard for open-ended generation tasks, but it is expensive and slow.

Perplexity: A Deeper Look

Perplexity deserves special attention because it is the most commonly reported metric for language models. It is defined as:

$$PPL = \exp( -(1/N) \Sigma \log P(token_i | context_i) )$$

Intuitively, a perplexity of k means the model is as uncertain as if it were choosing uniformly among k options at each step. A model with perplexity 10 is much more confident (and presumably more accurate) than one with perplexity 100. Code Fragment B.2 below puts this into practice.


# PyTorch implementation
# Key operations: loss calculation, training loop, results display
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

text = "The transformer architecture revolutionized natural language processing."
inputs = tokenizer(text, return_tensors="pt")

with torch.no_grad():
    outputs = model(**inputs, labels=inputs["input_ids"])
    perplexity = torch.exp(outputs.loss)
    print(f"Perplexity: {perplexity.item():.2f}")

Fun Note: The Perplexity Hall of Fame

GPT-2 (2019) achieved about 35 perplexity on WikiText-103. GPT-3 brought it below 20. Modern open models like LLaMA-3 push even lower. Each generation of models is, in a measurable sense, less surprised by human language. The trajectory suggests we are approaching fundamental limits of predictability in natural text.

Quick Reference Cheat Sheet

  • Supervised learning = labeled data; self-supervised = labels from structure; RL = learn from rewards.
  • Cross-entropy is the standard LLM loss function. AdamW is the standard optimizer.
  • Fight overfitting with weight decay, early stopping, and good data practices.
  • Use perplexity for language models, F1 for classification, ROUGE for summarization.
  • Automated metrics are useful but imperfect. Human evaluation and LLM-as-judge fill the gap.
  • Always check for data contamination before trusting benchmark results.
