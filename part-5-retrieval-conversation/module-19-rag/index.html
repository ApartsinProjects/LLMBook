Module 19

Retrieval-Augmented Generation (RAG)

Part V: Retrieval & Conversation

"The best answer is not always inside the model. Sometimes the smartest thing an AI can do is look it up."

A principle widely echoed across RAG practitioners, c. 2023

Chapter Overview

Large language models are powerful generators but inherently limited by their training data cutoff, their tendency to hallucinate, and the impossibility of encoding all world knowledge in model parameters. Retrieval-Augmented Generation (RAG) addresses these limitations by connecting LLMs to external knowledge sources at inference time, grounding responses in retrieved evidence rather than relying solely on parametric memory. Building on the embedding and vector database foundations from Module 18, RAG closes the gap between static model knowledge and dynamic, real-world information.

This module covers the complete RAG landscape, from fundamental architectures through advanced retrieval techniques. You will learn how to build ingestion pipelines, implement query transformations, combine dense and sparse retrieval, and leverage knowledge graphs for structured reasoning. The module also explores agentic RAG systems that can decompose complex queries, perform iterative research, and synthesize information from multiple sources.

On the structured data side, you will learn how LLMs can query databases through text-to-SQL, process tabular data, and combine structured and unstructured retrieval. Finally, the module surveys the major RAG frameworks (LangChain, LlamaIndex, Haystack) that provide production-ready tooling for building retrieval-augmented applications.

Practical Example: Stripe's Support Bot Hallucination Crisis

Who: The developer experience team at Stripe (2023).

Situation: Stripe deployed a GPT-4 powered support chatbot to help developers with API integration questions, relying solely on the model's parametric knowledge.

Problem: Within the first week, the bot confidently cited API endpoints that did not exist and fabricated parameter names, leading to frustrated developers filing spurious bug reports.

Dilemma: Pulling the bot entirely would waste months of work, but leaving it live risked eroding developer trust in Stripe's documentation ecosystem.

Decision: The team pivoted to a RAG architecture, indexing Stripe's entire API reference, changelog, and developer guides into a vector store.

How: Every user query was first embedded and matched against documentation chunks using cosine similarity. The top five chunks were injected into the prompt as grounding context, with instructions to cite sources and decline when no relevant documentation was found.

Result: Hallucinated API references dropped from roughly 18% of responses to under 2%. Developer satisfaction scores climbed back above the pre-bot baseline within a month.

Lesson: Parametric knowledge alone is dangerously unreliable for domain-specific facts. RAG transforms a confident guesser into a grounded assistant by forcing the model to show its sources.

Learning Objectives

  • Apply advanced retrieval techniques such as HyDE, multi-query expansion, cross-encoder re-ranking, and fusion retrieval (building on prompt engineering principles)
  • Construct and query knowledge graphs for structured reasoning, including GraphRAG with community detection
  • Build agentic RAG systems capable of query decomposition, iterative research, and multi-source synthesis
  • Implement text-to-SQL pipelines for structured data retrieval with schema linking and error correction
  • Evaluate RAG system quality using faithfulness, relevance, and answer correctness metrics
  • Compare and use RAG orchestration frameworks (LangChain, LlamaIndex, Haystack) for production applications
  • Diagnose and fix common RAG failure modes including lost-in-the-middle effects, retrieval drift, and context window overflow

RAG is essentially the AI equivalent of an open-book exam. The model still needs to understand the question and synthesize a good answer, but at least it does not have to pretend it memorized every fact in existence. Turns out, even language models do better when they can check their notes.

Practical Example: A Legal Tech Startup's Chunking Catastrophe

Who: The engineering team at a Series A legal tech startup building contract analysis tools (2024).

Situation: The team built a RAG system to answer questions about uploaded legal contracts, using a simple fixed-size chunking strategy of 512 tokens per chunk.

Problem: Critical contract clauses (indemnification, liability caps, termination triggers) frequently spanned chunk boundaries, causing the retriever to return incomplete or misleading fragments.

Dilemma: Increasing chunk size improved clause completeness but degraded retrieval precision and blew past context window limits when multiple documents were involved.

Decision: They implemented semantic chunking with overlap, splitting on paragraph and section boundaries rather than raw token counts, and added parent-child chunk relationships.

How: Each chunk stored metadata linking it to its parent section. When a child chunk was retrieved, the system optionally pulled the full parent section for context. Overlap of 15% between adjacent chunks ensured no clause was split without redundancy.

Result: Answer accuracy on clause-specific questions improved from 64% to 89%. False negatives (missing relevant clauses entirely) dropped by over 60%.

Lesson: Chunking strategy is not a minor implementation detail; it is a core architectural decision. Poor chunking silently corrupts your entire retrieval pipeline.

    • Practical Example: Notion's Hybrid Retrieval Breakthrough

    Who: The AI search team at Notion (2024).

    Situation: Notion was building an AI Q&A feature over users' workspaces, initially using pure dense (embedding-based) retrieval to find relevant pages and blocks.

    Problem: Dense retrieval excelled at semantic matching ("find notes about quarterly planning") but failed on exact keyword queries like project names, acronyms, and ticket IDs that users frequently searched for.

    Dilemma: Switching to BM25 keyword search would fix exact-match queries but destroy the semantic understanding that made the feature compelling in the first place.

    Decision: The team implemented fusion retrieval, running both BM25 and dense search in parallel and combining results with Reciprocal Rank Fusion (RRF).

    How: Each query triggered two parallel retrieval paths. BM25 scores and dense similarity scores were normalized, then merged using RRF with tunable weighting. A cross-encoder re-ranker made the final ordering.

    Result: Retrieval recall at k=10 improved by 23% compared to dense-only, and exact-match queries that previously returned zero results now succeeded over 95% of the time.

    Lesson: No single retrieval method dominates across all query types. Fusion retrieval with re-ranking captures the best of both semantic and lexical worlds.

    • Practical Example: McKinsey's Agentic RAG for Due Diligence

    Who: A digital transformation team at McKinsey building internal research tools (2024).

    Situation: Consultants performing due diligence on acquisition targets needed to synthesize information scattered across SEC filings, news articles, patent databases, and internal knowledge bases.

    Problem: A single due diligence question (e.g., "What regulatory risks does this company face in the EU?") required querying five or more sources, each with different formats and access patterns. Single-shot RAG returned shallow, incomplete answers.

    Dilemma: Building separate pipelines per source was expensive to maintain, but a monolithic index lost the structure and provenance that consultants required for their reports.

    Decision: They built an agentic RAG system that decomposed complex queries into sub-questions, routed each to the appropriate source, and iteratively refined its research until the answer met a completeness threshold.

    How: The agent used a planner LLM to break queries into sub-tasks, a router to dispatch each sub-task to the correct retrieval backend (vector store, SQL database, web search, or API), and a synthesizer that compiled results with full source citations.

    Result: Average research time per due diligence question dropped from 4 hours of manual work to 12 minutes of supervised AI research. Consultants reported that answer completeness matched or exceeded their own manual searches in 78% of cases.

    Lesson: Complex real-world questions rarely live in a single data source. Agentic RAG that can plan, route, and iterate is the bridge between toy demos and production research tools.

    Building a RAG system is a lot like organizing a library. You can have the smartest librarian in the world, but if the books are shelved randomly, the catalog is missing half the entries, and someone chunked the encyclopedias into individual paragraphs with no page numbers, even the librarian is going to hand you the wrong book. The retriever is only as good as the index it searches.

Bibliography

Foundational Papers

  1. Lewis, P., Perez, E., Piktus, A., et al. (2020). "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks." arxiv.org/abs/2005.11401

    The original RAG paper from Meta, introducing the paradigm of combining parametric (model) and non-parametric (retrieved) knowledge for generation.

  2. Gao, L., Ma, X., Lin, J., & Callan, J. (2022). "Precise Zero-Shot Dense Retrieval without Relevance Labels." (HyDE) arxiv.org/abs/2212.10496

    Introduces HyDE (Hypothetical Document Embeddings), using LLM-generated hypothetical answers as queries to improve zero-shot retrieval quality.

  3. Edge, D., Trinh, H., Cheng, N., et al. (2024). "From Local to Global: A Graph RAG Approach to Query-Focused Summarization." arxiv.org/abs/2404.16130

    Microsoft's GraphRAG paper, using knowledge graph community detection to answer global summarization queries that traditional RAG struggles with.

  4. Yan, S.Q., Gu, J.C., Zhu, Y., & Ling, Z.H. (2024). "Corrective Retrieval Augmented Generation." (CRAG) arxiv.org/abs/2401.15884

    Proposes a self-correcting RAG pipeline that evaluates retrieval quality and triggers fallback strategies when retrieved documents are irrelevant.

  5. Asai, A., Wu, Z., Wang, Y., Sil, A., & Hajishirzi, H. (2023). "Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection." arxiv.org/abs/2310.11511

    Trains a single LLM to decide when to retrieve, what to retrieve, and whether retrieved passages actually support its generated claims.

  6. Nogueira, R. & Cho, K. (2019). "Passage Re-ranking with BERT." arxiv.org/abs/1901.04085

    Demonstrates using cross-encoder BERT models for passage re-ranking, establishing the retrieve-then-rerank pattern standard in modern RAG pipelines.

Key Books

  1. Borgeaud, S., Mensch, A., Hoffmann, J., et al. (2022). "Improving Language Models by Retrieving from Trillions of Tokens." (RETRO) arxiv.org/abs/2112.04426

    DeepMind's RETRO architecture, integrating retrieval directly into the language model training process for trillion-token scale knowledge access.

  2. Liu, N.F., Lin, K., Hewitt, J., Paranjape, A., Bevilacqua, M., Petroni, F., & Liang, P. (2023). "Lost in the Middle: How Language Models Use Long Contexts." arxiv.org/abs/2307.03172

    Reveals that LLMs struggle to use information placed in the middle of long contexts, with implications for how retrieved documents should be ordered.

Tools & Libraries

  1. LangChain. github.com/langchain-ai/langchain

    Popular framework for building RAG applications, providing chains, retrievers, and integrations with vector stores, LLMs, and document loaders.

  2. LlamaIndex. github.com/run-llama/llama_index

    Data framework for connecting LLMs to external data sources, with purpose-built abstractions for indexing, querying, and retrieval pipelines.

  3. Haystack (deepset). github.com/deepset-ai/haystack

    Production-oriented NLP framework with modular pipeline architecture for building RAG, question answering, and semantic search applications.

  4. RAGAS. github.com/explodinggradients/ragas

    Evaluation framework for RAG pipelines, providing metrics for faithfulness, answer relevance, context precision, and context recall.
