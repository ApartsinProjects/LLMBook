Module 19 · Section 19.6

RAG Frameworks & Orchestration

Building production RAG applications with LangChain, LlamaIndex, and Haystack: chains, retrievers, query engines, and pipeline orchestration
★ Big Picture

Production RAG systems require far more than retrieval and generation. They need document loaders, text splitters, embedding integrations, vector store connectors, retriever abstractions, prompt templates, output parsers, evaluation harnesses, and deployment infrastructure. RAG frameworks like LangChain, LlamaIndex, and Haystack provide these building blocks as composable, well-tested components, dramatically reducing the engineering effort to go from prototype to production. Understanding their architectures, trade-offs, and best practices is essential for any RAG practitioner.

1. Why Use a RAG Framework?

Building RAG from scratch (as shown in earlier sections) is instructive but impractical for production. Real systems must handle dozens of document formats, integrate with multiple vector databases, support different LLM providers, manage conversation history, implement evaluation, and handle failure gracefully. RAG frameworks provide all of these as standardized, interchangeable components.

1.1 What Frameworks Provide

  • Document loaders: Connectors for PDF, HTML, DOCX, CSV, Notion, Confluence, Google Drive, S3, and dozens more.
  • Text splitters: Configurable chunking strategies (token-based, recursive character, semantic, markdown-aware).
  • Embedding integrations: Unified interfaces to OpenAI, Cohere, HuggingFace, and local embedding models.
  • Vector store connectors: Adapters for Chroma, Pinecone, Weaviate, Qdrant, Milvus, pgvector, and more.
  • Retriever abstractions: Dense, sparse, hybrid, ensemble, and contextual compression retrievers.
  • Chain/pipeline orchestration: Composable workflows that connect retrieval, augmentation, and generation steps.
  • Evaluation tools: Built-in metrics for retrieval quality, faithfulness, and answer relevance.

2. LangChain

LangChain is the most widely adopted RAG framework, providing a comprehensive toolkit for building LLM-powered applications. Its core abstraction is the chain: a composable sequence of operations that transforms inputs into outputs. For RAG, LangChain provides specialized retrieval chains that handle the full retrieve-augment-generate workflow.

2.1 Basic RAG with LangChain

from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# 1. Load and chunk documents
loader = PyPDFLoader("company_report.pdf")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ".", " "]
)
chunks = splitter.split_documents(docs)

# 2. Embed and store
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# 3. Build RAG chain with LCEL
prompt = ChatPromptTemplate.from_template("""
Answer the question based on the context below.
If you cannot answer from the context, say so.

Context: {context}

Question: {question}

Answer:""")

llm = ChatOpenAI(model="gpt-4o", temperature=0)

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {"context": retriever | format_docs,
     "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 4. Query
answer = rag_chain.invoke("What were the key findings?")

2.2 LangChain Expression Language (LCEL)

LCEL is LangChain's declarative composition syntax, using the pipe (|) operator to chain components together. Each component in the chain implements the Runnable interface, which provides invoke(), stream(), batch(), and ainvoke() methods. This uniform interface means that any component can be swapped without changing the pipeline structure.

ⓘ LCEL Benefits

LCEL provides several advantages over imperative chain construction: (1) automatic streaming support (the chain streams tokens as they are generated), (2) built-in async support (every chain automatically gets async methods), (3) optimized parallel execution (independent branches run concurrently), and (4) built-in tracing with LangSmith for debugging and evaluation. The pipe syntax also makes chain structure visually clear.

2.3 Advanced LangChain Retrievers

from langchain.retrievers import (
    ContextualCompressionRetriever,
    EnsembleRetriever
)
from langchain.retrievers.document_compressors import (
    CohereRerank
)
from langchain_community.retrievers import BM25Retriever

# Hybrid retrieval: dense + sparse
bm25_retriever = BM25Retriever.from_documents(chunks)
bm25_retriever.k = 10

dense_retriever = vectorstore.as_retriever(
    search_kwargs={"k": 10}
)

ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, dense_retriever],
    weights=[0.4, 0.6]
)

# Add re-ranking on top
compressor = CohereRerank(
    model="rerank-v3.5",
    top_n=5
)

reranking_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=ensemble
)

# Use in a chain
rag_chain = (
    {"context": reranking_retriever | format_docs,
     "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)

3. LlamaIndex

LlamaIndex (formerly GPT Index) takes a data-first approach to RAG, organizing its abstractions around indices and query engines rather than chains. Its core philosophy is that the way you structure and index your data determines the quality of your retrieval. LlamaIndex provides specialized index types optimized for different data structures and query patterns.

3.1 Index Types

  • VectorStoreIndex: The default index, storing chunk embeddings in a vector store for similarity search. Equivalent to the basic RAG pattern.
  • SummaryIndex: Creates summaries of documents and retrieves based on summary relevance. Good for questions that require high-level understanding.
  • TreeIndex: Builds a hierarchical tree of summaries, enabling top-down navigation from broad themes to specific details.
  • KnowledgeGraphIndex: Extracts entities and relationships to build an in-memory knowledge graph, enabling structured queries.
  • DocumentSummaryIndex: Combines document-level summaries with chunk-level retrieval for a two-stage approach.
from llama_index.core import (
    VectorStoreIndex,
    SimpleDirectoryReader,
    Settings
)
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding

# Configure global settings
Settings.llm = OpenAI(model="gpt-4o", temperature=0)
Settings.embed_model = OpenAIEmbedding(
    model="text-embedding-3-small"
)

# Load documents from a directory
documents = SimpleDirectoryReader("./data").load_data()

# Build vector index
index = VectorStoreIndex.from_documents(documents)

# Create query engine
query_engine = index.as_query_engine(
    similarity_top_k=5,
    response_mode="compact"  # or "tree_summarize", "refine"
)

# Query
response = query_engine.query(
    "What are the key findings from the report?"
)
print(response.response)
print(response.source_nodes)  # Retrieved chunks with scores

3.2 Query Engine Response Modes

LlamaIndex's query engines support multiple response synthesis strategies, each suited to different use cases. The response_mode parameter controls how retrieved chunks are combined into a final answer.

Response Mode Strategy Best For
compact Stuff all chunks into one prompt Short contexts, fast answers
refine Iterate through chunks, refining answer Many chunks, detailed answers
tree_summarize Recursively summarize chunks bottom-up Summarization tasks
simple_summarize Truncate context to fit, single call Quick summaries, cost-sensitive
accumulate Generate answers per chunk, combine Multi-faceted questions
★ Key Insight

The choice of response mode dramatically affects both quality and cost. compact is the cheapest (one LLM call) but may lose information if context is truncated. refine preserves all information by iterating through chunks but requires one LLM call per chunk. tree_summarize balances quality and cost by recursively summarizing in a tree structure. For most production use cases, compact with a generous token budget or tree_summarize for long documents provides the best trade-off.

4. Haystack

Haystack (by deepset) is a modular RAG framework built around the concept of pipelines as directed acyclic graphs (DAGs) of components. Unlike LangChain's chain abstraction or LlamaIndex's index abstraction, Haystack pipelines explicitly define data flow between components with typed inputs and outputs, making complex workflows more transparent and debuggable. 

from haystack import Pipeline
from haystack.components.retrievers.in_memory import (
    InMemoryEmbeddingRetriever
)
from haystack.components.builders import PromptBuilder
from haystack.components.generators import OpenAIGenerator
from haystack.document_stores.in_memory import (
    InMemoryDocumentStore
)

# Setup document store and retriever
doc_store = InMemoryDocumentStore()
retriever = InMemoryEmbeddingRetriever(
    document_store=doc_store,
    top_k=5
)

# Build pipeline as a DAG
template = """Answer based on the context.
Context:
{% for doc in documents %}
  {{ doc.content }}
{% endfor %}

Question: {{ question }}
Answer:"""

rag_pipeline = Pipeline()
rag_pipeline.add_component("retriever", retriever)
rag_pipeline.add_component("prompt", PromptBuilder(template=template))
rag_pipeline.add_component("llm", OpenAIGenerator(model="gpt-4o"))

# Connect components
rag_pipeline.connect("retriever.documents", "prompt.documents")
rag_pipeline.connect("prompt", "llm")

# Run
result = rag_pipeline.run({
    "retriever": {"query": "What are the key findings?"},
    "prompt": {"question": "What are the key findings?"}
})

6. Production Deployment Patterns

Moving a RAG system from prototype to production requires addressing several engineering concerns beyond the retrieval and generation logic: caching, monitoring, evaluation, and graceful degradation.

6.1 Common Production Patterns

  • Semantic caching: Cache responses for semantically similar queries to reduce latency and cost. If a new query is within a similarity threshold of a cached query, return the cached response.
  • Guardrails: Validate both inputs (block prompt injection attempts, off-topic queries) and outputs (check for hallucination, toxicity, PII leakage) before returning responses to users.
  • Observability: Log every step of the RAG pipeline (query, retrieved documents, prompt, response) for debugging and evaluation. Tools like LangSmith, Weights & Biases, and Phoenix provide tracing dashboards.
  • A/B testing: Run multiple RAG configurations simultaneously and measure which produces better user satisfaction, retrieval precision, and answer quality.
  • Fallback strategies: When retrieval returns no relevant results, fall back to the LLM's parametric knowledge with a clear disclaimer rather than returning an empty response.
⚠ Framework Lock-In

RAG frameworks evolve rapidly, with breaking API changes occurring frequently. To minimize lock-in: (1) wrap framework-specific code in thin adapter layers, (2) keep your core business logic (chunking strategy, prompt templates, evaluation criteria) independent of any framework, (3) use the framework for plumbing (document loading, vector store integration) but own your critical path, and (4) ensure your evaluation suite works independently so you can measure performance across framework migrations.

Section 19.6 Quiz

1. What are the main benefits of using a RAG framework instead of building from scratch?
Show Answer
RAG frameworks provide pre-built, well-tested components for document loading (dozens of formats), text splitting, embedding integration, vector store connectors, retriever abstractions, prompt management, output parsing, evaluation tools, and deployment infrastructure. They handle edge cases, provide streaming/async support, and integrate with observability tools. Building these from scratch would require months of engineering effort and ongoing maintenance.
2. How does LangChain's LCEL composition differ from Haystack's pipeline approach?
Show Answer
LCEL uses the pipe (|) operator to chain components in a linear or branching sequence, with implicit data flow between components through the Runnable interface. This is concise but can be opaque for complex pipelines. Haystack uses explicit DAG construction where components are added individually and connections are defined between named inputs and outputs. This requires more code but makes data flow transparent and easier to debug.
3. What are LlamaIndex's response modes, and when would you use "refine" versus "compact"?
Show Answer
Response modes control how retrieved chunks are synthesized into a final answer. "Compact" stuffs all chunks into a single prompt (one LLM call, fast and cheap, but may truncate context). "Refine" iterates through chunks one by one, refining the answer with each new chunk (one LLM call per chunk, preserves all information, but slower and more expensive). Use "compact" when context fits within the token budget and speed matters. Use "refine" when you have many chunks and need comprehensive answers that incorporate all retrieved information.
4. What is semantic caching and why is it important for production RAG?
Show Answer
Semantic caching stores responses indexed by the semantic similarity of queries, not exact string matching. If a new query is semantically similar (above a threshold) to a previously answered query, the cached response is returned. This is important because: (1) it dramatically reduces latency (cache hit vs. full RAG pipeline), (2) it reduces LLM API costs, (3) it provides consistent answers for similar questions, and (4) it reduces load on the vector database and LLM backend during traffic spikes.
5. How can you minimize framework lock-in when building a production RAG system?
Show Answer
Four strategies: (1) Wrap framework-specific code in thin adapter layers so the rest of your codebase depends on your interfaces, not the framework's. (2) Keep core business logic (chunking strategy, prompt templates, evaluation criteria) independent of any framework. (3) Use the framework for plumbing (document loading, vector store integration) but own your critical path. (4) Build your evaluation suite to work independently so you can measure performance before and after any framework migration.

Key Takeaways

  • Frameworks accelerate development enormously: LangChain, LlamaIndex, and Haystack provide battle-tested components for every stage of the RAG pipeline, from document loading to evaluation.
  • Choose the right abstraction for your needs: LangChain excels at complex agentic workflows with LCEL, LlamaIndex excels at data-heavy applications with specialized indices, and Haystack excels at transparent pipeline DAGs for production systems.
  • Production RAG requires more than a pipeline: Input/output guardrails, semantic caching, observability, A/B testing, and fallback strategies are essential for reliable production deployment.
  • Evaluation is non-negotiable: Use frameworks' built-in evaluation tools alongside independent test suites to continuously monitor retrieval quality, faithfulness, and answer relevance.
  • Plan for framework evolution: RAG frameworks evolve rapidly with frequent breaking changes. Minimize lock-in by owning your critical path and keeping business logic framework-independent.
