Working with LLM APIs

"The best interface is the one that disappears."

Pip AI Agent

Chapter Overview

Large language models are only as useful as the interface through which you access them. For the vast majority of production applications, that interface is an API: a set of HTTP endpoints exposed by OpenAI, Anthropic, Google, or an open-source serving framework. Knowing how to call these APIs correctly, efficiently, and reliably is a core skill for any engineer building with LLMs.

This chapter covers the full lifecycle of working with LLM APIs. We begin with the landscape of providers and their architectural differences, then move into structured output techniques and tool integration patterns that let models interact with external systems (a prerequisite for building AI agents). Finally, we tackle the engineering challenges of running LLM calls in production: routing across providers, caching, retry strategies, circuit breakers, cost management, and observability.

Practical Example: The Provider Outage That Saved a Product Launch

Who: A three-person backend team at a Series A healthtech startup building an AI-powered symptom checker.

Situation: The entire product relied on a single OpenAI API key with no fallback. Launch day was two weeks away.

Problem: OpenAI experienced a 90-minute outage during a critical investor demo, leaving the symptom checker completely non-functional. The CEO was mid-presentation when every request started returning 503 errors.

Dilemma: They could hard-code a second provider (fast but messy), or build a proper abstraction layer with automatic failover (cleaner but more time before launch).

Decision: They spent four days integrating LiteLLM with a fallback chain: OpenAI first, then Anthropic, then a self-hosted Mistral endpoint for emergencies.

How: They wrapped every LLM call in a provider-agnostic client, added health checks with circuit breakers, and configured Portkey as a gateway for routing and observability.

Result: During the next OpenAI degradation (three days after launch), the system silently switched to Anthropic. Zero user-facing errors. The investors never noticed.

Lesson: A single-provider architecture is a single point of failure. Building provider abstraction before you need it costs days; building it during an outage costs trust.

Practical Example: Taming the JSON Monster

Who: A senior ML engineer at an insurance company automating claims processing for 40,000 monthly submissions.

Situation: The team prompted GPT-4 to extract structured fields (claim type, date of incident, damage description, estimated cost) from free-text claim emails and return JSON.

Problem: About 12% of responses came back with malformed JSON: missing closing braces, invented field names, or numeric values wrapped in quotes. Each failure required manual review, creating a growing backlog.

Dilemma: They could add regex-based post-processing to patch common JSON errors (fragile), or enforce structure at the API level using response schemas and validation.

Decision: They switched to OpenAI's structured output mode with a strict JSON schema, then added Pydantic validation via the Instructor library as a second layer of defense.

How: They defined a Pydantic model for each claim type, configured Instructor to retry with corrective prompts on validation failure (up to three attempts), and logged every retry for analysis.

Result: Malformed output dropped from 12% to under 0.3%. The manual review backlog cleared within a week. Processing time per claim fell from 4 minutes to 45 seconds on average.

Lesson: Never trust free-form LLM output in a production pipeline. Enforce structure at the API layer and validate with schemas; the few extra lines of code save thousands of hours of manual cleanup.

Practical Example: The Semantic Cache That Cut Costs by 60%

Who: A platform engineering team at an edtech company serving 200,000 daily active students.

Situation: Students frequently asked near-identical questions ("What is photosynthesis?", "Explain photosynthesis", "How does photosynthesis work?"). Each query triggered a fresh API call to GPT-4, costing roughly $18,000 per month.

Problem: The team noticed that 40% of queries were semantically equivalent to questions asked in the previous 24 hours, yet every one generated a new API call and a new charge.

Decision: They implemented a semantic cache using embedding similarity. Incoming queries were embedded, and if a cached response existed within a cosine similarity threshold of 0.95, the cached answer was returned instantly.

How: They used Redis with a vector similarity index. Embeddings were generated with a lightweight model (text-embedding-3-small), and cache entries expired after 48 hours to keep answers fresh.

Result: API costs dropped from $18,000 to $7,200 per month. Median response latency fell from 2.1 seconds to 80 milliseconds for cache hits. Student satisfaction scores rose 11 points because answers felt instant.

Lesson: Before optimizing your prompts or switching models, check how many of your API calls are redundant. Semantic caching is often the highest-ROI optimization for high-volume LLM applications.

Practical Example: Streaming Saves a Customer Support Bot

Who: A product manager and two engineers at a SaaS company with 8,000 enterprise customers.

Situation: Their AI support bot generated thorough, multi-paragraph answers using Claude, but users experienced a 6 to 12 second blank screen while waiting for the full response.

Problem: User engagement data showed that 34% of customers abandoned the chat before the answer appeared. Internal surveys revealed that users assumed the bot was broken during the wait.

Dilemma: They could shorten answers (faster but less helpful), add a loading spinner (cheap but still felt slow), or implement streaming so tokens appeared in real time.

Decision: They implemented Server-Sent Events streaming, displaying tokens as they arrived. The first token appeared within 300 milliseconds.

How: They switched from the synchronous Messages API to the streaming variant, piped chunks through an SSE endpoint to the frontend, and added a typing indicator that transitioned smoothly into the streamed text.

Result: Abandonment rate dropped from 34% to 8%. Average session length increased by 40%. Customers rated the bot as "faster," even though total generation time was identical.

Lesson: Perceived latency matters more than actual latency. Streaming transforms a frustrating wait into a conversational experience, and the implementation cost is minimal compared to the user experience gain.
